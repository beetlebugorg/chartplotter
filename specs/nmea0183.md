# NMEA0183 connections & vessel state

NMEA0183 brings live navigation data — own-ship position, heading, AIS, depth,
wind, instruments — into the plotter. This spec defines how that data is
ingested, modelled, streamed, and rendered.

## Principle: a source, not a renderer

A NMEA0183 connection **renders nothing**. A single sentence stream carries
position, COG/SOG, heading, depth, wind, speed, AIS, and nav-to-waypoint — and
those fields fan out to *different* renderers. So ingestion is decoupled from
rendering:

```
[Connection: TCP client]  ──raw sentences──▶  NMEA0183 parser
   (v1; serial/udp/file later)                      │
                                           normalized vessel state
                                    (SignalK-shaped: navigation, environment,
                                            route, ais)
                                                     │
                                       stream to clients (SSE)
                                                     │
        ┌──────────────────────────┬─────────────────┴─────────────┐
   own-ship plugin             AIS overlay                 instruments / HUD
 (marker, COG vector,        (ais.vessels)              (depth, wind, STW, temp)
  follow camera)
```

The payoff: when NMEA2000 or SignalK is added later, it feeds the **same**
vessel-state model and every renderer is reused unchanged.

## Ingestion lives on the server

Consistent with the server-as-hub deployment model (the Pi is wired to the bus;
browser screens are synced displays):

- The most common modern path is a **multiplexer rebroadcasting NMEA0183 over
  TCP** on the boat network — so **TCP client is the only transport in v1**.
  Serial (USB), UDP listen, and File replay follow later behind the same
  `Source` abstraction.
- A browser cannot open raw TCP/UDP and Web Serial would bind data to one tab;
  ingestion must live where the wires are. Browsers stay pure displays
  downstream of the server stream.

## Server architecture

```
internal/engine/nmea/
  source.go        Source config + per-source goroutine (dial, read, reconnect, stats)
  manager.go       owns []Source, the VesselState, the AIS target map
  parse0183.go     sentence → model field mapping
  ais.go           VDM/VDO 6-bit payload decode → target map (scoped sub-piece)
  state.go         VesselState struct (mutex-guarded) + throttled delta emission
internal/engine/server/
  connections.go   /api/connections CRUD + persistence (clone of share.go)
  vessel.go        /api/vessel/stream (SSE) · /api/connections/<id>/raw (SSE sniffer)
```

### Source

Transport-agnostic config; v1 only constructs `tcp-client` inputs.

```go
type Source struct {
    ID        string
    Name      string
    Transport string // "tcp-client" (v1); "serial"|"udp"|"file" later
    Host      string
    Port      int
    Protocol  string // "nmea0183"
    Direction string // "in" (v1); "out"|"both" later (autopilot RMB/APB)
    Enabled   bool
}
```

One goroutine per enabled source: dial → read line-framed sentences → feed
parser → update state. Auto-reconnect with backoff. Each source tracks live
stats that drive the UI health badges:

```go
type SourceStatus struct {
    State      string    // "connecting"|"connected"|"stale"|"error"|"disabled"
    LastRx     time.Time
    RateHz     float64   // EWMA of sentences/sec
    Errors     int
    Sentences  []string  // sentence types seen (RMC, VDM, DPT…)
    Talkers    []string  // talker IDs seen (GP, AI, II…)
}
```

Persistence mirrors `share.go`: sources stored in `<dataDir>/connections.json`,
edited via `/api/connections`.

### Libraries

- `github.com/adrianmo/go-nmea` — instrument sentence set.
- `github.com/BertoldVdb/go-ais` — AIVDM/AIVDO 6-bit payload decode.

## Vessel-state model

One mutex-guarded struct, shaped after SignalK's vessel tree so future sources
map cleanly. Fields are pointers / option types so "unknown" is distinct from
"zero".

```
navigation:
  position{lat, lon}            GGA, GLL, RMC
  cogTrue                       VTG, RMC
  sog                           VTG, RMC
  headingTrue                   HDT
  headingMagnetic               HDG, HDM
  magneticVariation             HDG, RMC
  rateOfTurn                    ROT
  speedThroughWater             VHW
  datetime                      ZDA, RMC

environment:
  depth.belowTransducer         DBT
  depth.belowKeel               DBK
  depth.belowSurface            DPT, DBS
  water.temperature             MTW
  wind.angleApparent            MWV (R)
  wind.speedApparent            MWV (R)
  wind.angleTrue                MWV (T)
  wind.speedTrue                MWV (T), MWD
  wind.directionTrue            MWD

route:                          (nav-to-active-waypoint)
  xte                           XTE, RMB, APB
  bearingToWaypoint             RMB, BWC, APB
  distanceToWaypoint            RMB, BWC
  activeWaypoint                RMB, APB

ais:
  vessels[mmsi]{position, cog, sog, heading, name, type, …}   VDM/VDO
                (TTL-evicted when stale)
```

### Sentence → field coverage (v1, full instrument)

| Sentence | Feeds |
|---|---|
| GGA, GLL, RMC | position |
| VTG, RMC | cogTrue, sog |
| HDT / HDG / HDM | headingTrue / headingMagnetic + variation |
| ROT | rateOfTurn |
| VHW | speedThroughWater |
| DPT / DBT / DBK / DBS | depth.* |
| MWV / MWD | wind.* (apparent + true) |
| MTW | water.temperature |
| ZDA | datetime |
| RMB / APB / XTE / BWC | route.* |
| VDM / VDO | ais.vessels |

> **AIS is a scoped sub-piece.** VDM/VDO are multi-fragment, 6-bit-packed, with
> ~6 relevant message types (1/2/3 position, 5 static, 18/19 class B, 24). It is
> in the model from day one but isolated in `ais.go` so it can't block the
> straightforward instrument parsing.

## Streaming to clients

Reuse the SSE pattern proven by `import.go` job events.

- **`GET /api/vessel/stream`** — coalesced vessel-state deltas. Throttle by
  field class so a 10 Hz feed doesn't flood every screen: position ~1–2 Hz,
  instruments ~4 Hz.
- **`GET /api/connections`** — list + live `SourceStatus`. Status changes are
  also pushed on the stream so badges update without polling.
- **`POST/PUT/DELETE /api/connections`** — CRUD, persisted to `connections.json`.
- **`GET /api/connections/<id>/raw`** — live raw-sentence SSE for the sniffer.

Client side, a single **`VesselStateStore`** (mjs) subscribes to the stream and
is the one shared source of truth. Render plugins read from it and never touch
the server directly.

## Rendering (downstream consumers)

| Plugin | Reads | Renders |
|---|---|---|
| own-ship | `navigation.*` | marker, heading/COG vector, `updateFollow` + `setCameraMode` |
| instruments / HUD | depth, wind, STW, temp | numeric readouts (minimal/stub ok in v1) |
| AIS overlay | `ais.vessels` | target overlay |

own-ship is the first visible payoff; the others can land incrementally on the
already-populated model.

## Connections UI

A "Connections" tab in the existing settings dialog: a list of sources with live
health badges fed by the status stream.

```
┌─ Connections ─────────────────────────────────────┐
│  ● Multiplexer      tcp 10.0.0.20:2000          ▶  │  ● green  data flowing
│      RMC GGA VTG VDM DPT MWV · 9.2 Hz · 0 errors   │  ⚠ amber  stale / parse errs
│  ⚠ AIS receiver     tcp 10.0.0.20:2001          ▶  │  ◌ grey   disabled
│      no data 47s                                   │
│  [ + Add connection ]                              │
└────────────────────────────────────────────────────┘
```

"Add connection" is a one-step v1 form (only one transport): **Name · Host ·
Port**; protocol NMEA0183 and direction In are fixed. Each row expands to a
**raw sniffer** (subscribes to `/api/connections/<id>/raw`) — the single most
useful tool when debugging real boat wiring, and it sets up File-replay as the
next transport.

## TODO

- **Smooth own-ship motion.** GPS fixes arrive ~1 Hz, so the own-ship marker
  jumps each fix. Dead-reckon between fixes on the client — extrapolate position
  from the last fix using COG/SOG (and heading) on each animation frame, snapping
  to the next real fix when it arrives — for fluid movement. Same idea can smooth
  AIS targets. Keep it purely visual (don't fabricate data the picker reports).
