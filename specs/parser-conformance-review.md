### [s57/feature (FRID/FOID/ATTF/FFPT)] ATTF attribute value parsing incorrectly terminates on 0x1F instead of using field terminator
- **Code:** `internal/s57/parser/feature.go:142-153`
- **Spec:** 31Main.pdf clause 4.4 and 7.6.3 table 7.22; clause 2.5 field and subfield termination
- **Issue:** The parseAttributes function searches for 0x1F (1/15 subfield terminator per ISO 8211) as the value terminator (lines 142-144). However, per S-57 spec clause 4.4, the ATVL subfield 'must be a string of characters terminated by the subfield terminator (1/15)' which is correct for ASCII mode. But in BINARY mode (implied by this parser reading binary FRID), the entire ATTF field is terminated by the ISO 8211 field terminator (typically 0x1E), not individual ATVL by 0x1F. The code mishandles this: it assumes every attribute value ends at 0x1F, but should instead parse ATTF as repeating ATTL (2 bytes, b12 int) + ATVL (variable, gt = general text up to 0x1F in ASCII or variable-length in binary). Per clause 2.5, the ATVL in BINARY is general text 'gt' which has no specified terminator within the repeating group - termination is by field boundary. The code's logic of finding 0x1F as a terminator is ASCII-mode-only and will fail or misparse binary data with no 0x1F present, or with premature 0x1F in the middle of the attribute value.
- **Fix:** Implement proper ATTF field parsing: (1) Determine if data is ASCII or binary from context. (2) For ASCII: parse as repeating [ATTL(2 bytes text) + ATVL(text up to 0x1F)] until field ends. (3) For binary: parse as repeating [ATTL(2 bytes, b12 LE uint) + ATVL(variable-length text or bytes, terminated by 0x1F per spec 7.6.3)] or until field end. (4) Implement proper subfield and field boundary detection per ISO 8211 clause 2.5.
- **Verifier:** Per spec clause 2.5 (31Main.pdf p3.10), field and subfield termination: All S-57 fields (ISO/IEC 8211 data fields) must be terminated by the 'Field Terminator' (FT) which is 0x1E (1/14) at lexical level 0. Subfield terminators are 0x1F (1/15) at lexical level 0. Code line 142-144 treats 0x1F as the end of each attribute value, but this is only correct if in ASCII mode with UTF encoding. For binary mode, the ATTF field boundary itself (0x1E) terminates all attributes within it. Code does not distinguish ASCII vs binary implementation mode and incorrectly assumes 0x1F terminates individual ATVL values universally. Per clause 7.6.3 table 7.22, ATVL is 'A( )' which means variable-length and format-dependent.

### [s57/updates (RUIN/RVER apply)] INSERT offset calculation violates S-57 spec
- **Code:** `internal/s57/parser/updates.go:357-373`
- **Spec:** S-57 Part 3, clause 8.4.2.4 (page 3.56), 8.4.3.2 (page 3.58), 8.4.3.3 (page 3.59): INSERT instruction position semantics
- **Issue:** The S-57 spec states for INSERT: 'The insertion must start at the position one before the index specified by the FSIX/VPIX/CCIX subfield.' This means if the 1-based index is N, insertion occurs at 0-based position N-2. The code performs `idx := ...Uint16(...) - 1` for all instruction types (line 358), which correctly converts 1-based to 0-based for DELETE and MODIFY (which specify absolute positions) but is wrong for INSERT. For INSERT, this causes items to be inserted one position too far into the array. Example: when FSIX=2 (insert at position 1 in 1-based), the spec intends 0-based index 1, but the code calculates idx=1 and inserts at 0-based index 1, which places items after the intended position instead of before it, corrupting feature-to-spatial pointer sequences and edge node references.
- **Fix:** Differentiate INSERT from DELETE/MODIFY in index calculation. For INSERT only, use `idx := int(binary.LittleEndian.Uint16(ctrl[off+1:off+3])) - 2` to account for 'position one before'; DELETE and MODIFY retain `- 1`.
- **Verifier:** S-57 Part 3, clause 8.4.2.3 (p3.56), 8.4.2.4 (p3.57), and 8.4.3.3 (p3.59) all state for INSERT: 'The insertion must start at the position one before the index specified by the [FFIX/FSIX/CCIX/VPIX] subfield.' The index subfields are 1-based. If FSIX=2 (1-based), 'position one before' means position 1 (1-based) = index 0 (0-based). Code at line 358: `idx := Uint16(...) - 1` converts to 0-based as idx=1, then line 364-373 inserts BEFORE idx, placing items at index 1 (0-based) = position 2 (1-based). This is one position too far. Correct formula for INSERT only: `idx := Uint16(...) - 2`.
